## Stage 1 总结

### 优化效果
- **大 C shape (S1/S8)**：tiling 带来 3.5-3.9× 加速，通过减少 C 的 DRAM 写回
- **小 shape (S2/S4/S7)**：tiling 无收益甚至有害，指令开销 > cache 收益
- **列主序**：所有 kernel 远弱于行主序 (1-3 GFLOPS vs 6-13)。原因：列主序 naive 已有 stride-1，tiling 增量小
- **最佳结果**: naive row S4 = 13.04 GFLOPS (17.5% of peak per core)

### 后续方向
- 多线程并行：S1/S8 计算量大，适合并行；S2/S6 太小不适合
- loop order 优化：列主序需要重新设计 tile 拆分。后续仅优化行主序
- 行主序 tiled loop 的 1t 性能 (8-12 GFLOPS) 是后续 OMP 的 baseline

## Stage 2 Step 1 分析

### 关键发现
1. **S1 可扩展**: 64t 达到 511 GFLOPS，42× 加速。96t 饱和（m/Ti=63 < 96）
2. **S8 完全无扩展**: 1t=10.97, 96t=9.72 GFLOPS。Ti×n strip = 576 MB，内存瓶颈
3. **小 m shape (S2/S3/S5/S6/S8) 只有 1 tile**: Step 1 本质不可并行
4. **1t 性能优于串行 tiled**: OMP 版本重新排列了 tile loop order (ii→pk→jj vs pk→ii→jj)，减少了循环开销

### 瓶颈
- **并行度不足**: m/Ti 太小（S2=1, S3=1, S5=1, S6=1, S8=1）
- **Strip 太大**: Ti×n 不能 fit L2，C 反复 miss
- **列主序放弃**: 行主序性能已达列主序 10×+

### 后续方向
- **collapse(2)** 并行化 jj 维度 → 每个线程 Ti×Tj tile (128 KB)，并行度 = (m/Ti)×(n/Tj)
- **线程私有累加区** → 减少 false sharing
- **k 维并行** → 对 S6 (巨型 k) 方向

# 理论分析 — 最优 Kernel & Tile 选择

## 1. Roofline 模型

先理解每个 shape 的瓶颈类型，才能选对优化方向。

### 计算瓶颈 vs 访存瓶颈

算术强度 AI = FLOP / Byte：

| Shape | m | n | k | AI | 理论峰值 GFLOPS (1 core) | 所需带宽 (GB/s) |
|-------|---|---|---|------|------------------------|------------------|
| S1 | 4000 | 160000 | 128 | 30.98 | 74.56 | **2.41** |
| S2 | 8 | 16 | 16000 | 1.33 | 74.56 | **55.93** |
| S3 | 32 | 16000 | 16 | 2.67 | 74.56 | **27.99** |
| S4 | 144 | 144 | 144 | 12.00 | 74.56 | **6.21** |
| S5 | 16 | 12344 | 16 | 2.00 | 74.56 | **37.31** |
| S6 | 4 | 64 | 606841 | 0.94 | 74.56 | **79.18** |
| S7 | 442 | 193 | 11 | 2.54 | 74.56 | **29.34** |
| S8 | 40 | 1127228 | 40 | 5.00 | 74.56 | **14.91** |

一台核的 L1 带宽 ~128 GB/s，L2 ~64 GB/s，DRAM (单通道) ~25 GB/s。

- **AI > 10** → 计算瓶颈（S1, S4）：FMA 单元限制，cache 命中率至关重要
- **AI < 3** → 访存瓶颈（S2, S3, S5, S6, S7）：内存带宽限制，需要减少 DRAM 流量
- **AI ≈ 5** → 均衡（S8）：两种瓶颈都有

### 多线程下的 Roofline 变化

当线程数增加时，对 DRAM 带宽的需求线性增长。以 S1 为例：

| 线程数 | 所需 DRAM 带宽 (GB/s) | 单 NUMA 带宽上限 |
|--------|----------------------|-------------------|
| 1 | 2.41 | ~80 (含 L3 缓存效应) |
| 24 | 57.8 | ~80 |
| 48 | 115.7 | **超出** 1 NUMA (80) |
| 96 | 231.3 | **需跨 4 NUMA** (320) |

这就是为什么 48+ 线程时需要 `--interleave=all` 的 NUMA 策略。

## 2. Kernel 选择决策树

### 决策公式

给定 shape (m,n,k) 和线程数 Nt，按以下逻辑选择 kernel：

```
第一步：并行度分析
  C_tiles = ceil(m/Ti) × ceil(n/Tj)    # S2 的并行度
  k_iters  = ceil(k/Tk)                 # S4 需要的 pk 迭代数

第二步：Kernel 选择
  if C_tiles == 1:
      if k_iters >= Nt:  → S4_3d    # pk 并行可覆盖所有线程
      else:               → S3_3d    # 1 个 tile，单线程就够了
  elif C_tiles > 1000:
      if Nt >= 48:        → S3_3d    # 大 C，private buffer 减少 false sharing
      else:               → S2_3d    # 中低线程，collapse(2) 开销更低
  else:                   → S2_3d    # tile 不多，S2 最轻量
```

### 特殊情况

- **S2 (m=8,n=16,k=16000)**: C_tiles = 1, k_iters = 63。Nt ≤ 16 时 S4_3d 最优（pk 分给 16 线程 = 每线程 4 轮）；Nt > 16 时 S4 的 idle 线程太多，S3_3d 回退。
- **S6 (m=4,n=64,k=606841)**: C_tiles = 1, k_iters = 2372。S4_3d **总是**最优——2372 轮分给 96 线程 = 每线程 25 轮，充分饱和。
- **S8 (m=40,n=1127228,k=40)**: C_tiles = 4404, k_iters = 1。S2_mn 最优——MN-only 减少一层循环，Ti=40 (全 m) 减少 barrier 次数。

## 3. Tile 大小选择理论

### 3.1 L2 Cache 约束

一个 C tile (Ti×Tj) + 一个 B 条带 (Tk×Tj) 必须 fit L2：

```
Ti × Tj × 8 + Tk × Tj × 8  ≤  L2_size  = 1 MB
```

等价于：
```
Tj × (Ti + Tk) × 8  ≤  1,048,576
Tj × (Ti + Tk)  ≤  131,072
```

### 3.2 寄存器约束

AVX-512 有 32 个 ZMM 寄存器 (每个 64 字节 = 8 doubles = 512 位)。微内核的 register tile (tr×tc)：
- C: tr × tc doubles → 需要 tr × ceil(tc/8) 个 ZMM
- B: ceil(tc/8) 个 ZMM（可复用）
- A broadcast: 1 个 ZMM

对于 tr=8, tc=16：C = 8×2 = 16 ZMM，B = 2 ZMM，A = 1 ZMM → 使用 19/32 ZMM。

可选更大的 register tile：
- **8×24**: C = 24 ZMM, B = 3 ZMM, A = 1 → 28/32 ZMM（紧张但可行）
- **6×32**: C = 24 ZMM, B = 4 ZMM, A = 1 → 29/32 ZMM

### 3.3 并行度约束

collapse(2) 的并行度必须 ≥ Nt 才能避免 idle 线程：

```
ceil(m/Ti) × ceil(n/Tj)  ≥  Nt
```

### 3.4 汇总公式

给定 (m,n,k,Nt) 和 kernel 类型：

```
Ti ∈ [8, m]   且 Ti 是 tr 的倍数
Tj ∈ [16, n]  且 Tj 是 tc 的倍数
Tk ∈ [k, ...] 且 Tk ≤ k (无意义超过 k)

约束:
  ① 并行度:    ceil(m/Ti) × ceil(n/Tj) ≥ Nt   (S2/S3 only)
  ② L2 fit:    Tj × (Ti + Tk) × 8 ≤ 1,048,576
  ③ k 饱和:    k/Tk ≥ Nt/4                     (S4 only)
  ④ 对齐:      Ti % tr = 0, Tj % tc = 0
```

2. **软件预取**: 在 micro_kernel 内提前 4 轮 `_mm_prefetch` B 和 A → 预期 +10%
3. **Nontemporal store**: C 写回用 `_mm512_stream_pd` 跳过 cache → 减少 eviction → 预期 +5%
4. **最优 Ti**: 当前 Ti=64, Ti=32 并行度更高且 tile fit L1 → 预期 +5%


不需要外层 tiling, 不需要 private buffer, 不需要 collapse。k=16000 全程在寄存器累加, 最后一次性写回 128B。

- k=144, Tk=144: pk=1 → S4 无用

### 为什么 S4 的 1t 比 S5 更好

S4 只有 2 tiles, S3/S2 的 collapse 在 4t 后收益衰减。AVX-512 微内核在 1t 达到 56.1 GFLOPS (75% peak) → 接近理论极限。

### 后续方向 (S6d)

1. **AVX-512 全矩阵 naive**: 不 tiling, 1 线程 AVX-512 扫全 144³ (fit L3) → 简单, 高效
2. **Register tile 12×16**: 当前 8×16, 扩展到 12×16 (24 ZMM for C) → 多 50% reuse
3. **多线程优化**: 当前最优是 4t, 但 tile 太少 → 可以用 S4 (k-parallel) 做 144 轮 pk → 扩展性更好

**目标**: 150+ GFLOPS multi-thread (2× per-core vs S5 4t)

## S6: (4, 64, 606841) — 极小 C, 极大 k, Bandwidth-bound

### 优化历程

| Stage | 最优 kernel | GFLOPS | 关键突破 |
|-------|------------|--------|----------|
| Serial naive | — | **8.4** | scalar naive 最优 (C=128B) |
| S2-S2 | s2_3d | 8.4 | 1 tile, 无扩展 |
| S2-S4 | s4_3d | **192** | **23×**: k=606841→2372 pk iters→并行化 |
| S5 | avx512_s5 | 7.2 | **严重退步**: AVX-512 gather TLB thrash |

### Shape 特征分析

- C=2KB: 完全可放 L1
- k=606841: 极大 k。Tk=256→2372 pk rounds → **S4 天然优势**
- B=k×n=606841×64×8=310MB: 每次遍历 k 要读全部 B
- A=k×m=606841×4×8=19.4MB: 也很大
- AI=0.94: **极端访存密集** — 每 1 次 FMA 需要 1 byte 数据

### 为什么 AVX-512 退步

AVX-512 每次加载 512-bit (64 bytes) B。但 B 的 310MB 远超 TLB coverage (L2 TLB=1536 entries × 4KB=6MB)。每个 `_mm512_loadu_pd` 可能触发 TLB miss → 硬件走页表 → 几百个 cycle 停顿。

标量版 naive p-i-j 只有 stride-1 的 B[p*n+j] → 硬件预取器能提前加载下一行 → TLB miss 被隐藏。

### 后续方向 (S6f) — 最大潜力

1. **B packing (关键)**: 把 B 按 tile 重排为连续内存 → `packed_B[k_idx][j_idx]` → TLB miss 降 90%+
2. **Huge pages (2MB/1GB)**: B 用大页分配 → TLB 覆盖翻 512× (4KB→2MB)
3. **A packing**: A 也需要 packing (19MB), 同理
4. **Nontemporal loads from packed B**: B 只读一次 → 用 `_mm512_stream_load_si512` 跳过 cache

**预期**: 300+ GFLOPS (S4 k-parallel + packed B + AVX-512 continuous load)

## S8: (40, 1127228, 40) — 极大 n, 小 m/k, Bandwidth-bound

### 优化历程

| Stage | 最优 kernel | GFLOPS | 关键突破 |
|-------|------------|--------|----------|
| Serial tiled_3d | — | 8.0 | cache tiling 必须 |
| S2-S1 (Ti×n OMP) | — | 11.0 | **无扩展**: 1 tile |
| S2-S2 (collapse) | s2_mn | 549 | **50×**: collapse(2) 4404 tiles |
| S2-S3 (priv buf) | s3_mn | 394 | **退步**: buf 分配 4404× |
| S5 (AVX-512) | avx512_s5_omp | **664** | **+21%** over S2 |

### Shape 特征分析

- C=40×1.1M×8=360MB: cache tiling 必须
- n=1127228: 极大。B strip = Tk × Tj。B 跨 n 步长 → TLB miss 严重
- m=40, Ti=40→m/Ti=1: 只有 jj 方向 4404 tiles
- k=40, Tk=40→pk=1: S4 无效
- AI=5: 计算-访存均衡

### 为什么 S3 退步而 S5 进步

S2 直接写 C, 4404 tiles × 8 bytes = 35KB per tile。S3 每 tile 分配/清零 128KB buf→4404 次 = overhead > 收益。S5 的 AVX-512 内层加速抵消了 buf 开销 (写回次数减少 8× via 向量化)。

### 后续方向 (S6h)

1. **B packing (关键)**: B 40×1.1M=360MB。Packing B[pk:k_end][jj:jj+Tj] → stride-1 + TLB coherent → 预期 +30%
2. **Nontemporal store for C writeback**: C 写回不走 cache → 减少 eviction
3. **Ti=m (40) 全用**: 减少 tile loop 开销
4. **Tj 调优**: 当前 Tj=256。Tj=512 减少 tile 数→减少 collapse overhead

**目标**: 900+ GFLOPS
